### 先说结论：推荐系统正在经历什么

过去十年，几乎所有大厂推荐系统的结构都一样：

```text
亿级候选池
  → 召回（向量检索，万级）
    → 粗排（轻模型，千级）
      → 精排（重模型，百级）
        → 重排（策略调整，十级）
          → 展示
```

这套架构在数据驱动的时代跑赢了一切，但它有四个越来越致命的问题：

**1\. 优化目标割裂**：召回、粗排、精排各自优化不同的目标函数，互相掣肘，全局最优永远无法达到。

**2\. 前链路制约后链路上限**：召回阶段漏掉的 item，精排模型再强也无力回天——一旦过滤，机会永远消失。

**3\. GPU 算力白白浪费**：传统 CTR 模型（DIN/DCN/DLRM 等）继承自 CPU 时代，大量算子是 memory-bound，MFU（模型浮点运算利用率）通常只有 **4-5%**。你花百万买的 H100，实际只用了 5%。

**4\. 跟不上 LLM 时代的技术红利**：Scaling Law、MoE、[RL 对齐](https://zhida.zhihu.com/search?content_id=274239025&content_type=Article&match_order=1&q=RL+%E5%AF%B9%E9%BD%90&zhida_source=entity)……LLM 那边每隔几个月就有革命性突破，传统推荐架构完全无法承接。

**2024-2025 年，快手和抖音给出了各自的破局方案：**

-   **[快手 One系列](https://zhida.zhihu.com/search?content_id=274239025&content_type=Article&match_order=1&q=%E5%BF%AB%E6%89%8B+One%E7%B3%BB%E5%88%97&zhida_source=entity)**：颠覆式——用一个端到端生成式模型，直接替换整个多阶段系统
-   **[抖音 RankMixer](https://zhida.zhihu.com/search?content_id=274239025&content_type=Article&match_order=1&q=%E6%8A%96%E9%9F%B3+RankMixer&zhida_source=entity)**：渐进式——重新设计精排架构，让判别式模型也能享受 Scaling Law

接下来逐一拆解。

### Part 1：快手 One系列全景

### 1.1 技术起点：[QARM](https://zhida.zhihu.com/search?content_id=274239025&content_type=Article&match_order=1&q=QARM&zhida_source=entity)（2024.11）

**论文**：QARM: Quantitative Alignment Multi-Modal Recommendation at Kuaishou
**arXiv**：2411.11739

要理解 [OneRec](https://zhida.zhihu.com/search?content_id=274239025&content_type=Article&match_order=1&q=OneRec&zhida_source=entity)，先要理解 QARM 解决的问题——**多模态推荐为什么那么难用？**

工业界做多模态推荐的常见方式：

```text
视频内容（标题/封面/音频）
        ↓ 预训练好的多模态大模型（MLLM）
   固定的 Embedding
        ↓ 推荐模型额外输入
   CTR/排序预估
```

听起来合理，但有两个根本问题：

**问题1：Representation Unmatching（表示不匹配）**

MLLM 是用 NLP/CV 任务训练的，”相似”的定义是语义相似。但推荐系统里的”相似”是**行为相似**——同样喜欢”苹果水果”的人，和喜欢”苹果手机”的人完全不同，但 MLLM 可能认为它们相关。

**问题2：Representation Unlearning（表示无法更新）**

预训练好的 MLLM 输出会被缓存下来作为固定输入，推荐模型的梯度**无法反传**到多模态模型——多模态表示永远停在原地，无法随推荐任务持续优化。

QARM 的解法：**两步走**——

**Step 1：Item Alignment（用推荐任务的知识教 MLLM）**

挖掘推荐场景下真正”相似”的 item pair（通过历史 U2I/I2I 检索），用对比学习微调 MLLM，让它的表示空间与推荐对齐。

**Step 2：Quantitative Code（量化为可学习的 Semantic ID）**

把对齐后的连续 Embedding 用 K-Means 量化为离散 token（VQ 或 RQ），作为推荐模型的输入——现在梯度可以通过 STE（Straight-Through Estimator）反传，多模态模型终于可以端到端学习。

**线上效果**：广告 Revenue **+9.704%**，电商 GMV **+2.296%**。

> QARM 是快手从”多模态辅助推荐”走向 OneRec 的关键技术桥梁。

### 1.2 核心论文：OneRec（2025.02）

**论文**：OneRec: Unifying Retrieve and Rank with Generative Recommender and Iterative Preference Alignment
**arXiv**：2502.18965

这篇是快手 One系列的**奠基论文**，也是工业推荐历史上的一个里程碑。

**核心思想：把推荐变成”写作文”**

传统推荐是**判别式**的：给 1000 个候选 item 逐一打分，取分最高的。

OneRec 是**生成式**的：根据用户历史行为序列，直接”生成”用户接下来想看的内容列表。

就像 GPT 根据上文续写下文，OneRec 根据用户的历史行为 token 续写下一批推荐内容。

**三大技术创新：**

**① Semantic ID：视频如何变成 token**

不能把传统 ID 当 token——传统 ID 之间没有语义关系（ID 1 和 ID 2 不代表内容相似）。

OneRec 用 **RQ-Kmeans**（Residual K-Means 量化）把每个视频编码为 3 个有语义层次的 token：

```text
Level 1：[0-511]  → 粗粒度语义（如：搞笑类 / 美食类 / 科技类）
Level 2：[0-511]  → 中粒度语义（大类内细分）
Level 3：[0-511]  → 细粒度语义（精细区分）

视频 A = [12, 847, 3291]
视频 B = [12, 851, 4107]  ← L1 token 相同，说明 A/B 是同类
```

为什么不用 RQ-VAE（传统方案）？RQ-VAE 有严重的 **codebook collapse（沙漏现象）**——大量 token 从不被使用，实际有效码本极小。RQ-Kmeans 强制每个 cluster 包含等量 item，彻底解决此问题。

**② Session-wise 生成（vs Point-wise 生成）**

之前的生成式推荐（如 TIGER）逐个预测下一个 item，item 之间独立。

OneRec 直接生成**完整的 session**（一次请求的整个推荐列表），列表内各 item 相互感知，更符合真实场景的 session 一致性。

**③ IPA：迭代偏好对齐**

NTP 预训练只能学历史分布——历史数据来自旧系统，学到的天花板就是旧系统的上限。

推荐系统的 RL 有个特殊难点：**没有对照组**。NLP 的 RLHF 可以让人对比两个回答打分，但推荐系统每次请求只有一次曝光机会，无法同时生成两个版本让用户比较。

OneRec 的解法：

1.  训练一个奖励模型（RM）来预测用户满意度
2.  用 beam search 生成多个候选 session，RM 打分
3.  取最优/最差构建偏好对，DPO 优化
4.  **迭代执行**，逐步超越历史系统天花板

**线上效果**：

-   快手 App 停留时长 **+0.54%**，快手极速版 **+1.24%**
-   7日用户生命周期 LT7 **+0.05%/+0.08%**
-   全部核心指标（点赞/关注/评论/广告）同时正向

> 在快手体系里，0.1% 停留时长提升具有统计显著性。1.24% 是什么概念？相当于每天多留住用户好几分钟。

### 1.3 工程报告：OneRec Technical Report（2025.06）

**arXiv**：2506.13695

如果说 2502.18965 是”我证明了这件事可行”，那 2506.13695 是”我告诉你我是怎么做到的”。

**三大工程发现：**

**① 推荐系统的 Scaling Law 存在**

参数规模从 0.015B 扩到 2.633B，训练损失持续稳定下降，业务指标同步提升。Scaling Law 在推荐场景成立——但前提是**架构要能充分利用 GPU**。

技术报告给出了三维 Scaling 路径：

-   **Feature Scaling**：扩展特征规模
-   **Codebook Scaling**：扩大 Semantic ID 码本
-   **Infer Scaling**：推理阶段加大 beam size

**② 10× FLOPs，MFU 从 ~5% → 23.7%/28.8%**

通过类 LLM 的 Encoder-Decoder 架构，关键算子数量压缩 **92%**（从约 15,000 个 → 1,200 个），不必要的通信和存储开销大幅降低。

整体运营成本（OPEX）仅为传统推荐 Pipeline 的 **10.6%**。

**③ RL 技术在端到端架构上释放出前所未有的价值**

在传统多阶段系统里，RL 很难落地（奖励信号传播路径太长）。在 OneRec 的端到端框架下，RL 的效果非常显著。

### 1.4 架构演进：OneRec-V2（2025.08）

**arXiv**：2508.20900

**核心升级：Encoder-Decoder → [Lazy Decoder-Only](https://zhida.zhihu.com/search?content_id=274239025&content_type=Article&match_order=1&q=Lazy+Decoder-Only&zhida_source=entity)**

OneRec v1 采用 Encoder-Decoder 结构。V2 提出 **Lazy Decoder-Only 架构**，移除独立 Encoder，简化 Cross-Attention，用单一 Decoder-Only 结构统一处理用户历史和生成任务。

效果惊人：

| 指标 | v1 (Enc-Dec) | V2 (Lazy Dec-Only) |
| --- | --- | --- |
| 计算量 | 基线 | 减少 94% |
| 实际训练资源 | 基线 | 减少 90% |
| 可支持参数量 | 0.5B | 8B（16× 扩张） |

为什么叫”Lazy”？Decoder 以”懒惰”的方式处理历史序列——不再用专门 Encoder 压缩，直接拼入 Decoder 上下文，简洁高效。

**RL 升级：IPA → ECPO（Early-Clipped GRPO）**

V1 的 IPA（迭代 DPO）在某些情况下训练不稳定。V2 提出 ECPO：对负优势（A < 0）样本进行更严格的梯度截断，在保留样本的同时防止梯度爆炸。

### 1.5 推理能力：OneRec-Think（2025.10）

**arXiv**：2510.11639

**问题**：OneRec 系列虽然推荐效果好，但本质上还是个”黑盒”——输入历史行为，输出推荐列表，没有可解释的推理路径。

**类比 LLM 的突破**：DeepSeek-R1 等模型通过引入 CoT（思维链）推理，让模型”想一想再回答”，效果大幅提升。能不能把这个思路搬到推荐？

**OneRec-Think 的方案**：统一集成对话 + 显式推理链 + 个性化生成推荐的单一模型：

```text
用户历史行为 + 对话指令
      ↓ Think（显式推理）
"用户最近看了很多美食内容，
 但偏好偏向家常菜而非高档餐厅，
 结合地理位置判断可能关注本地菜谱..."
      ↓ 生成推荐列表
[视频 A, 视频 B, 视频 C, ...]
```

推理链不仅让结果更准确，还让推荐可解释、可交互（用户可以说”不对，我不喜欢辣的”然后系统实时调整）。

**工业部署关键：Think-Ahead 架构**

推理生成成本高、延迟长，无法直接用于实时推荐。Think-Ahead 将推理分为两阶段：

-   **离线**：预计算用户兴趣推理摘要
-   **在线**：基于摘要快速解码推荐结果

实现了推理能力与实时延迟的兼顾。

### 1.6 开源基础模型：[OpenOneRec](https://zhida.zhihu.com/search?content_id=274239025&content_type=Article&match_order=1&q=OpenOneRec&zhida_source=entity)（2025.12）

**arXiv**：2512.24762

快手将 OneRec 技术栈完整开源，构建**通用推荐基础模型**（Foundation Model for Recommendation）。

核心贡献：

-   推荐领域数据 + 通用领域数据**联合预训练**
-   **Itemic-Text Alignment**：统一 item token 和自然语言的语义空间
-   **RecIF-Bench**：4 层 8 类任务的推荐专属评估体系（从语义理解到推荐理由生成）

面向学术界和中小企业开放，推动”生成式推荐”成为新的社区范式。

### Part 2：抖音 RankMixer（2025.07）

**论文**：RankMixer: Scaling Up Ranking Models in Industrial Recommenders
**arXiv**：2507.15551
**机构**：ByteDance（抖音）

快手走的是”颠覆式”路线，直接替换整个系统。抖音走的是**渐进式**路线：不动现有 Pipeline，只重新设计精排模型，让判别式推荐也能 Scaling。

### 为什么传统精排无法 Scaling？

**两大障碍**：

**① 推理延迟限制**：工业精排必须在几十毫秒内返回结果，参数量不能无限增加。

**② CPU 时代遗留架构**：DIN、DCN、FM 等特征交叉模块是为 CPU 设计的，充满 memory-bound 算子，无法充分利用 GPU 的大规模并行能力。结果就是：你买了 H100，MFU 只有 **4.5%**，剩下 95.5% 的算力在空转。

参数量翻 10 倍，推理成本翻 10 倍，但效果没有 10 倍提升——Scaling Law 在这里失效了。

### RankMixer 的解法：硬件感知架构

**两个核心模块**：

**① Multi-Head Token Mixing（替代 Self-Attention）**

传统 Self-Attention 的复杂度是 O(N²)，且包含大量内存访问。Token Mixing 用共享权重的线性混合替代，复杂度 O(N)，全矩阵乘法，**GPU 并行度极高**。

**② Per-Token FFN（替代共享 FFN）**

传统方案所有 token 共享同一个 FFN，特征之间争抢同一个参数空间。Per-Token FFN 给每个特征一个独立的 FFN，**参数容量大幅提升，但计算并行**。

**MFU 提升效果**：

| 系统 | MFU |
| --- | --- |
| 传统推荐精排（DIN/DCN 等） | 4.5% |
| OneRec（快手）训练 | 23.7% |
| OneRec（快手）推理 | 28.8% |
| RankMixer（抖音） | ~45% |

**Sparse MoE：8× 参数，成本不变**

通过 **DTSI + ReLU Routing**，参数量从 100M 扩到 1B（8×），但推理吞吐量反而提升 >50%：

-   DTSI 动态决定激活哪些 Expert，解决 Expert 训练不足
-   ReLU 路由天然稀疏，不需要手动限制激活数

**线上效果（抖音全量部署）**：

-   活跃天数 **+0.3%**（抖音体量下，数千万用户）
-   App 使用时长 **+1.08%**
-   广告 ADVV **+3.90%**

推理延迟：RankMixer-1B 与原来的 16M 基线模型**持平**。

### 两条技术路线，各有胜负手

最后做个横向对比：

| 维度 | 生成式（快手 OneRec） | 判别式（抖音 RankMixer） |
| --- | --- | --- |
| 核心假设 | 端到端生成能超越多阶段 | 精排模型本身就能 Scaling |
| 对现有系统影响 | 替换整个 Pipeline | 只替换精排 |
| MFU | 23.7% ~ 28.8% | ~45% |
| 优点 | 全局最优，LLM 技术直接迁移 | 改造成本低，风险小 |
| 难点 | 工程重构复杂 | Scaling 收益有上限（召回仍是瓶颈） |
| 适合场景 | 底层架构重建 | 现有系统持续优化 |

**两条路线不是非此即彼**。在一个成熟的大厂推荐系统里，生成式与判别式可以共存：精排继续优化（RankMixer 路线），同时探索用生成式模型逐步替换召回+粗排阶段（One系列路线）。

**个人判断**：中长期看，生成式推荐的天花板更高——端到端优化是推荐系统走向 AGI 技术栈的必经之路。但判别式 Scaling（RankMixer 路线）是现阶段收益最确定、风险最低的选择。
