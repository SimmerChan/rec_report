## 快手向左，阿里向右｜2025[大模型](https://zhida.zhihu.com/search?content_id=274010139&content_type=Article&match_order=1&q=%E5%A4%A7%E6%A8%A1%E5%9E%8B&zhida_source=entity)融入推荐系统的两条路线深度解读

**一句话概括：**2025年下半年，推荐系统领域出现了一场意义深远的路线分叉——快手选择用大模型的「思考能力」彻底重写推荐系统，用 [CoT 链式推理](https://zhida.zhihu.com/search?content_id=274010139&content_type=Article&match_order=1&q=CoT+%E9%93%BE%E5%BC%8F%E6%8E%A8%E7%90%86&zhida_source=entity)替换传统排序逻辑；阿里选择让大模型「预言未来兴趣」，以[生成式预训练](https://zhida.zhihu.com/search?content_id=274010139&content_type=Article&match_order=1&q=%E7%94%9F%E6%88%90%E5%BC%8F%E9%A2%84%E8%AE%AD%E7%BB%83&zhida_source=entity)为传统CTR模型注入血液。两种方案都在工业线上拿到了真实的业务增益，但代表的是截然不同的技术哲学——一个是颠覆，一个是渐进。

本文将对两条路线做完整的技术梳理和横向对比，并分享个人对这场路线之争未来走向的判断。

## 一、背景：LLM 进入推荐，需要解决什么问题？

在正式展开两个方案之前，先明确推荐系统融合大模型面临的核心矛盾。

传统工业推荐系统的排序阶段是一条成熟的流水线：离线特征工程 → 精排 CTR/CVR 模型 → 在线毫秒级推断。在这条流水线里，「推理」这件事是被刻意去掉的——模型只需做确定性的打分，任何涉及语言理解或多步推理的操作都意味着无法接受的延迟。

大模型的进入打破了这个平衡。LLM 带来的能力不只是更大的参数量，而是三件事：

-   **语义理解：**能把用户历史行为序列理解成有意义的兴趣叙事，而不只是一串 ID；
-   **推理能力：**能沿着逻辑链做多步思考，而不只是一次前向传播打个分；
-   **生成能力：**能生成「未来」——不管是直接生成推荐列表，还是预测用户兴趣的稠密表征。

但这三件事中，哪个才是推荐系统最需要的？快手和阿里给出了不同的答案。

## 二、快手路线：[OneRec-Think](https://zhida.zhihu.com/search?content_id=274010139&content_type=Article&match_order=1&q=OneRec-Think&zhida_source=entity)——让推荐模型学会「思考」

### 2.1 动机：推荐系统需要 CoT 吗？

快手 OneRec-Think 的出发点有些反直觉：大模型在数学、编程、逻辑推理上的突破，来源于「Chain-of-Thought（链式思维推理）」——让模型在给出最终答案前，先经历一个显式的思考过程，把中间步骤写出来。

问题来了：推荐有逻辑推理的需求吗？

快手团队的答案是：有，而且一直被忽视。他们指出，推荐决策本质上是一个多步骤的理解过程：先理解用户的长期兴趣偏好 → 结合当前上下文（时间、场景、心情信号） → 判断候选内容与当前需求的语义匹配度 → 考虑多样性和惊喜感 → 最终排序。传统 [CTR 模型](https://zhida.zhihu.com/search?content_id=274010139&content_type=Article&match_order=1&q=CTR+%E6%A8%A1%E5%9E%8B&zhida_source=entity)把这些步骤压缩成一次前向传播，中间没有任何可解释的推理链，所以很难处理需要「绕几个弯」才能判断的场景——比如，用户最近看了很多运动视频，但今天搜了「冥想」，应该推运动恢复类内容还是纯冥想内容？

### 2.2 OneRec-Think 三阶段设计

OneRec-Think 采用了三阶段训练方案，逐步让推荐模型具备真正的推理能力：

**第一阶段：语义对齐（Semantic Alignment）**

从零开始，把推荐系统的核心概念「喂给」大模型：用户行为序列、物品属性、候选池。这一阶段目标是让 LLM 的语言空间和推荐空间完成初步对齐——让模型知道「CTR 是什么」「用户行为序列的时序含义」「物品的多粒度属性表示」。

**第二阶段：推理激活（Reasoning Activation）**

用构造的推理链数据进行监督微调，激活模型在推荐场景下进行 CoT 推理的能力。这里的推理链不是凭空生成的——团队从历史推荐日志中挖掘出高质量的「用户为什么喜欢这个内容」的标注数据，构造出显式的推理路径作为训练信号。

**第三阶段：推理增强（Reasoning Enhancement）**

引入强化学习，以推荐效果（用户实际点击、观看完成率等）作为奖励信号，对推理链的质量和最终推荐决策做端到端优化。这一步让模型不只是「会说正确的推理」，而是「能推理出更好的推荐」。

### 2.3 工程上的挑战与取舍

CoT 最大的问题是延迟。一条思维链可能包含几百个 token，推理时间在秒级，根本无法直接用于在线排序（工业要求毫秒级）。

快手的解法是分层推理架构：

-   离线/近线层：LLM 对每个用户进行周期性「深度思考」，生成该用户在各类场景下的兴趣推理摘要（稠密向量化）；
-   在线层：轻量模型使用离线生成的推理向量做实时排序，延迟回到毫秒级。

这让 CoT 推理的语义深度和工业部署的延迟约束之间找到了平衡点。

### 2.4 线上 A/B 结果

在快手主 APP 的线上 A/B 实验中：

-   **APP 停留时长 +0.159%**（快手体量下这是显著的增益）；
-   互动指标（点赞、评论、分享）全面正向；
-   用户多样性指标有小幅提升，说明推理机制确实在一定程度上抑制了信息茧房效应。

## 三、阿里路线：[AMEN](https://zhida.zhihu.com/search?content_id=274010139&content_type=Article&match_order=1&q=AMEN&zhida_source=entity)——让大模型「预言未来兴趣」

### 3.1 动机：不颠覆，而是增强

阿里的 AMEN（Autoregressive Modeling of Evolving Next-interest）框架出发点截然不同。它的核心逻辑是：现有的工业 CTR 模型经过多年打磨，在延迟、稳定性、工程可维护性上都是经过验证的，直接替换风险极高。那能不能让大模型只做「增量」的工作，把它的能力以特征的形式注入现有排序体系，而不是替换整个架构？

AMEN 的答案是：让 LLM 预测用户「未来兴趣流（[Next Interest Flow](https://zhida.zhihu.com/search?content_id=274010139&content_type=Article&match_order=1&q=Next+Interest+Flow&zhida_source=entity)）」，把这个预测结果作为额外特征喂给传统 CTR 模型。

### 3.2 Next Interest Flow 是什么？

传统推荐模型对用户兴趣的建模是「回顾性」的：用户过去点击了什么，我就以此推断他喜欢什么。这种方式对兴趣漂移（用户兴趣随时间变化）的捕捉能力弱，特别是对用户「即将要产生」的兴趣几乎是盲的。

AMEN 定义了「Next Interest Flow」——用户未来一段时间内兴趣演化方向的稠密向量序列表示。这不是一个标量分数，而是一个连续的、多维度的「兴趣预测轨迹」，描述了用户在不同内容类别、不同情感色调、不同信息深度上的兴趣走向。

关键创新在于：AMEN 用自回归语言模型（类 GPT 的架构）对这个「兴趣流」做生成式预训练——把用户历史行为序列当做「上文」，让模型预测「下文」（未来兴趣），训练目标是最小化预测兴趣流和真实未来行为的重建误差。

### 3.3 与 CTR 模型的集成

预训练完成后，AMEN 产出每个用户的「Next Interest Flow 向量」，以如下方式注入传统排序体系：

-   作为用户侧的新增稠密特征，直接拼入 CTR 模型的输入层；
-   或作为注意力引导信号，在序列建模层对候选物品做偏置。

整个改造对现有排序系统的侵入性极低——不需要改模型主干，不需要重新设计推理管线，只是新增了一路特征。这让线上部署风险降到最低。

### 3.4 线上 A/B 结果

在淘宝信息流上的线上实验：

-   **GMV 提升 11.24%**（高得有些惊人，通常信息流 GMV 能稳定 +1% 就值得全量）；
-   CTR 和 CVR 均有正向提升；
-   由于对现有系统改动极小，风险回退也非常简单。

## 四、两条路线的横向对比

### 4.1 技术哲学的差异

快手路线的核心假设是：**推荐本质上是一个需要推理的任务**，传统排序模型只是因为「学不会推理」才不得不用大量特征工程来打补丁。如果真正解决了推理问题，整个架构可以变得更优雅。

阿里路线的核心假设是：**推荐系统的价值来自对用户兴趣动态的捕捉**，而大模型最擅长的恰好是对序列的生成建模。最优解不是颠覆现有系统，而是用大模型来填补传统模型在「兴趣预测」上的盲区。

前者是「换引擎」，后者是「加油助力」。

### 4.2 适用场景的差异

| 维度 | 快手 OneRec-Think | 阿里 AMEN |
| --- | --- | --- |
| 架构改动程度 | 大（重建推荐主链路） | 小（特征级注入） |
| 部署风险 | 高 | 低 |
| 能力覆盖 | 语义理解 + 多步推理 | 兴趣预测 + 动态建模 |
| 延迟解法 | 离线预计算 + 轻量线上 | 预训练特征离线存储 |
| 对新物品/冷启动 | 语义泛化强 | 依赖历史行为序列 |
| 线上核心指标 | 停留时长 +0.159% | GMV +11.24% |

### 4.3 共同的工程智慧

虽然技术路线不同，两个方案都得出了相同的工程结论：**大模型不能直接端到端上线做实时推荐**。无论是 CoT 推理还是生成式预训练，计算密集的部分都被推到了「离线」或「近线」层，线上只执行轻量的推断或特征查表。

这个架构选择并非无奈妥协，而是当前阶段的最优解——它让大模型的深度计算和工业系统的极低延迟之间找到了平衡，也给了工程团队更可控的 A/B 实验窗口。

## 五、一个更大的视角：LLM 重定义推荐系统的三个层次

把 OneRec-Think 和 AMEN 放在更大的视野里，2025年大模型融入推荐系统的演进实际上有三个层次：

**第一层：特征层（2024年的主流）**——用 LLM 提取语义特征，替换或增强 item embedding。代表工作：[PatchRec](https://zhida.zhihu.com/search?content_id=274010139&content_type=Article&match_order=1&q=PatchRec&zhida_source=entity)、PAD、字节 RARE 等。改动最小，风险最低，但能力提升有限。

**第二层：表征层（2025年上半年）**——用 LLM 做更长期的用户建模，预测兴趣动态（如 AMEN），生成式预训练提升表征质量。是当前工业界落地最活跃的层次。

**第三层：决策层（2025年下半年探索中）**——用 LLM 直接参与推荐决策链，不只是提取特征，而是做推理、生成候选、乃至取代传统排序（如 OneRec-Think、OneRec-V3）。这一层最激进，工程代价最高，但天花板也最高。

快手选择直接跳到第三层，阿里在第二层做到了极致。谁更对？这取决于公司的技术储备、风险偏好和业务节奏——并没有一个普适的答案。

## 六、对从业者的启示

1.  **「生成式预训练 + 特征注入」是最低风险的落地路径。**如果你所在的团队需要快速验证大模型对推荐效果的提升，AMEN 范式（离线生成用户表征 → 在线注入现有模型）是当前工程代价最小、验证最快的方式，值得作为第一步尝试。
2.  **CoT 推荐的核心挑战不是模型，而是训练数据。**OneRec-Think 三阶段中最难的部分是第二阶段的推理链数据构造——如何从海量用户日志中挖掘出高质量的「用户理解推理链」，是这个方向的关键卡点。
3.  **「离线计算 + 在线轻推断」是未来一段时间的标准架构。**不要奢望 LLM 能直接在推荐排序的关键路径上做实时推理，至少在当前算力和延迟约束下不现实。工程重心在于如何把 LLM 的深度计算高效地转化为可以低延迟访问的离线资产。
4.  **关注 RLHF / GRPO 在推荐里的应用。**阿里淘宝搜索已经把 DeepSeek R1 的 GRPO 后训练技术降维到向量召回，直接用 GMV 等商业指标作为奖励函数。这条路线如果规模化成功，将是比特征注入更深层的范式升级。

## 七、结语

快手和阿里的技术路线，某种程度上映射的是两种工程文化的差异：快手更愿意赌一个激进的愿景，接受更高的技术风险；阿里更倾向于用最小的改动撬动最大的确定性收益。

但两条路的目的地是一样的：让推荐系统真正理解用户，而不只是统计用户过去点了什么。

从这个角度看，这场路线之争不是零和游戏——它们在不同的技术层次上共同推动着推荐系统向「真正智能」迈进。很可能在未来某一天，「离线 CoT 推理 + 生成式兴趣预测 + 轻量在线排序」会成为统一的工业标准架构。
